# 提案資料スライド連動ロールプレイ機能 - 要件定義書

## 意図分析サマリー

- **ユーザーリクエスト**: NPCとの会話中に提案資料（PDF）のスライドを表示し、意図的に提示したスライドのみをAIに渡してNPC応答に反映させる
- **リクエストタイプ**: New Feature（新機能追加）
- **スコープ**: Multiple Components（フロントエンド会話画面、シナリオ作成画面、NPC会話エージェント、バックエンドAPI、S3ストレージ）
- **複雑度**: Complex（PDF処理、マルチモーダルAI連携、UI変更、データ構造変更）

---

## 1. 機能要件

### FR-001: 提案資料アップロード
- シナリオ作成画面に新しいステップ「提案資料」を追加する
- 既存のPDFアップロード機能（Knowledge Base連携用）とは別管理とする
- 対応形式: PDF形式のみ
- 1シナリオにつき1ファイルまで添付可能
- 提案資料は任意（オプション）。資料なしのシナリオも従来通り動作する

### FR-002: PDF→スライド画像変換
- アップロードされたPDFを各ページごとに画像（PNG）に変換する
- 変換タイミング: アップロード時にバックエンドで事前変換
- 変換された画像はS3に保存し、CloudFront経由で配信する
- 各ページの画像URLとページ番号をDynamoDBに保存する

### FR-003: 会話画面スライドビューアー（下部トレイ方式）
- 会話画面の下部にスライドサムネイルトレイを配置する
- アバターは中央に大きく表示を維持する
- トレイにはスライドのサムネイル一覧を横スクロールで表示する
- 現在選択中のスライドはハイライト表示する
- 提示済みスライドには✓マークを表示する
- 🔍ボタンでスライドの拡大モーダルを表示する（人間が読めるサイズ）
- 拡大モーダル内でもスライド送り（◀▶）が可能
- 提案資料がないシナリオではトレイを非表示にする

### FR-004: スライド提示判定（手動ボタン方式）
- 「📋 提示」ボタンをユーザーが明示的にクリックすることでスライドをAIに提示する
- スライド切り替えの途中で一瞬だけ表示されたスライドはAIに渡さない
- 提示済みスライドは視覚的にマーク（✓）する
- 一連のトークの中で複数のスライドを提示可能
- 同じスライドの再提示も可能とする

### FR-005: スライド情報のAI連携（画像Vision方式）
- 提示されたスライドの画像をBedrock Vision（マルチモーダル）でNPC会話AIに渡す
- トークとスライドを明確に分離した構造でAIに渡す
- メッセージごとに「このメッセージ送信時に提示したスライド番号リスト」を付与する
  - データ構造例: `{ message: "...", presentedSlides: [3] }`
- NPC会話エージェントのプロンプトにスライドコンテキストセクションを追加する
- NPCはスライドの内容を踏まえた応答を生成する

### FR-006: スライド提示履歴の記録
- どのスライドをいつ提示したかの履歴をセッションデータに記録する
- 記録項目: スライド番号、提示タイムスタンプ、提示時のメッセージID

### FR-007: スライド提示のフィードバック統合
- セッション分析時にスライド提示履歴を評価に含める
- 評価観点例:
  - スライド提示のタイミングの適切さ
  - 提示したスライドと会話内容の関連性
  - 提示順序の論理性
- フィードバックレポートにスライド活用の評価セクションを追加する

## 2. 非機能要件

### NFR-001: パフォーマンス
- PDF→画像変換: アップロード後30秒以内に完了
- スライド画像のロード: 初回表示2秒以内（CloudFrontキャッシュ活用）
- 拡大モーダルの表示: 0.5秒以内
- スライド画像のAI送信によるレイテンシー増加: +1秒以内

### NFR-002: ストレージ
- スライド画像はS3に保存（既存のPDFストレージとは別バケット or 別プレフィックス）
- 画像形式: PNG（解像度: 1920x1080推奨）
- セッション終了後もスライド画像は保持（シナリオに紐づくため）

### NFR-003: セキュリティ
- 提案資料のアクセスはCognito認証必須
- S3署名付きURLによるアクセス制御
- シナリオの公開設定に応じたアクセス制御

### NFR-004: アクセシビリティ
- スライドトレイはキーボード操作対応（矢印キーでスライド選択）
- 提示ボタンはaria-label付与
- 拡大モーダルはEscキーで閉じる、フォーカストラップ対応
- スクリーンリーダーでスライド番号・提示状態を読み上げ可能

### NFR-005: 国際化
- 新規UIテキストは日英両言語対応（i18nキー使用）

## 3. 段階的実装計画

### Phase 1: スライド表示 + 手動提示
- 提案資料アップロード（シナリオ作成画面の新ステップ）
- PDF→画像変換パイプライン（Lambda + S3）
- 会話画面の下部スライドトレイUI
- 拡大モーダル
- 「提示」ボタンによる手動提示
- 提示済みマーク表示

### Phase 2: AI連携 + フィードバック統合
- NPC会話エージェントへのスライド画像送信（Bedrock Vision）
- プロンプトへのスライドコンテキスト追加
- メッセージ×スライド紐付けデータ構造
- スライド提示履歴の記録
- セッション分析へのスライド評価統合
- フィードバックレポートへのスライド活用評価追加

## 4. 技術的決定事項

| 項目 | 決定 | 理由 |
|------|------|------|
| ファイル形式 | PDF形式のみ | pdf.jsでブラウザ内レンダリング可能、既存機能活用、Lambda処理容易 |
| スライド提示判定 | 手動ボタン | 誤送信防止、ユーザーの完全制御、実装シンプル |
| AI連携方式 | 画像Vision | 図表・グラフの内容もAIが理解可能、NPC応答品質が最も高い |
| UIレイアウト | 下部スライドトレイ | アバターを最大限大きく表示、サムネイルで全体把握、🔍で拡大確認 |
| 資料管理 | 既存PDFとは別ステップ | Knowledge Base連携用PDFとの混同を防止 |
| ファイル数上限 | 1ファイル/シナリオ | シンプルな運用、1提案資料で1ロールプレイ |
| 提案資料の必須性 | 任意（オプション） | 従来の会話のみシナリオも継続サポート |
| フィードバック | スライド提示履歴を評価に含める | タイミング・関連性・順序の評価 |
| データ構造 | メッセージ×スライド番号紐付け | トークとスライドの明確な分離 |
| 実装段階 | Phase 1（表示+提示）→ Phase 2（AI連携+フィードバック） | 段階的にリスク低減 |
