# 並列コーディングワークフロー

## 概要
コーディングタスクを実施する際は、事前に設計とタスク分割を行い、SubAgentを活用して並列実装することで効率を最大化する。

## ワークフロー

### Phase 1: 設計（Design）
コーディング開始前に以下を明確化する：

1. **影響範囲の特定**: 変更が必要なファイル・コンポーネントを洗い出す
2. **依存関係の分析**: ファイル間・コンポーネント間の依存関係を把握する
3. **変更方針の決定**: 各ファイルに対する具体的な変更内容を決定する
4. **インターフェース定義**: コンポーネント間の型定義・APIインターフェースを先に確定する

### Phase 2: タスク分割（Task Decomposition）
設計結果に基づき、並列実行可能なタスクに分割する：

1. **独立性の確認**: 各タスクが他のタスクに依存せず単独で実行可能であることを確認する
2. **タスク粒度の調整**: 1タスク = 1〜3ファイルの変更を目安とする
3. **共有リソースの事前処理**: 型定義・共通ユーティリティなど共有リソースは並列タスク開始前に完了する
4. **タスク仕様の明確化**: 各タスクに対して以下を明記する：
   - 変更対象ファイル
   - 具体的な変更内容
   - 入出力のインターフェース（型定義）
   - 参照すべき既存コード
   - 守るべきコーディング規約

### Phase 3: 並列実装（Parallel Implementation）
SubAgent（`general-task-execution`）を使用して並列実装する：

1. **事前準備タスクの実行**: 共有型定義・インターフェース変更など依存される変更を先に実施する
2. **並列タスクの同時実行**: 独立したタスクをSubAgentで同時に実行する
3. **SubAgentへの指示**: 各SubAgentに以下を提供する：
   - 変更対象ファイルのパス
   - 具体的な変更仕様（何をどう変えるか）
   - 関連する型定義・インターフェース情報
   - プロジェクトのコーディング規約（日本語コメント、i18n対応等）
4. **contextFilesの活用**: SubAgentに関連ファイルをcontextFilesとして渡し、コンテキストを共有する

### Phase 4: 統合・検証（Integration & Verification）
並列実装完了後に統合と検証を行う：

1. **型チェック**: getDiagnosticsで変更ファイルの型エラーを確認する
2. **整合性確認**: コンポーネント間のインターフェースが正しく接続されていることを確認する
3. **リント実行**: `cd frontend && npm run lint` でリントエラーがないことを確認する
4. **修正対応**: エラーがあれば修正する

## 適用条件

### 並列実装を適用する場合
- 3ファイル以上の変更が必要な場合
- 変更対象ファイルが互いに独立している場合
- 各タスクの変更仕様が明確に定義できる場合

### 逐次実装で行う場合
- 1〜2ファイルの小規模変更
- ファイル間の依存が強く、順序が重要な場合
- 探索的な変更（変更内容が事前に確定できない場合）

## SubAgent呼び出しのベストプラクティス

```
invokeSubAgent:
  name: "general-task-execution"
  prompt: |
    以下のタスクを実行してください：
    
    ## タスク概要
    [具体的な変更内容]
    
    ## 変更対象
    - ファイル: [パス]
    - 変更内容: [詳細]
    
    ## インターフェース仕様
    [型定義・API仕様]
    
    ## コーディング規約
    - コメントは日本語で記述
    - 変数名・関数名は英語
    - i18nキーを使用（日英対応）
    - TypeScript型定義を厳密に行う
    - anyの使用を避ける
  contextFiles:
    - path: [関連ファイル1]
    - path: [関連ファイル2]
```

## 注意事項
- SubAgentは同一ファイルを同時に編集しないこと（競合防止）
- 共有リソース（型定義等）の変更は必ず並列タスク開始前に完了すること
- SubAgentの実行結果は必ず検証し、不整合があれば修正すること
